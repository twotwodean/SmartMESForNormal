# 시각화·장비표현·상태 애니메이션 갭분석 및 구현계획 v1.0

> 문서 성격: **조사·감사·계획 문서**. 본 문서는 코드를 변경하지 않으며, 이후 구현 단계의 입력물이다.
> 범위: SmartMES의 시각적 표현력(설비/라인 상태 표시, 애니메이션, 공장 배치도, Andon, 트렌드 차트 등)을 실제 상용 MES/SCADA 제품과 비교하고, 이 코드베이스(Next.js 14 + Tailwind + SSE + 엄격한 CSP)에 맞는 구체적 구현 순서를 제시한다.

---

## 1. 요약

**현재 수준 한 줄 요약**: SmartMES는 SSE로 실제 데이터가 초 단위로 갱신되는 견고한 실시간 백엔드를 갖췄지만, 화면은 카드·배지·정적 원형 게이지로만 표현되어 "값은 실시간, 화면은 정지화면"인 상태다 — 값이 바뀌어도 눈에 띄는 시각적 신호(깜빡임, 펄스, 부드러운 전이)가 전혀 없고, 라인/설비를 공간적으로 배치한 도면(플로어맵)이나 장비 아이콘, Andon 보드, 흐름 애니메이션이 전혀 없다.

**핵심 갭**:
1. **상태 애니메이션 부재** — `ALARM`(알람) 상태조차 정지된 빨간 배지 하나로만 표시되고 점멸(blink)하지 않는다. 값 변경 시 flash 효과 없음. 게이지는 값이 바뀌면 CSS 전이 없이 순간 점프한다.
2. **공간적 표현(플로어맵) 부재** — 설비가 그리드의 카드로만 나열되고, 실제 라인 배치(공정 순서, 물리적 위치)를 반영한 2D 도면이 없다.
3. **장비 아이콘/심볼 부재** — 모든 설비가 텍스트 카드로 동일하게 표현되며, CNC/컨베이어/로봇 등 장비 유형을 구분하는 시각 기호가 없다.
4. **Andon 보드, 컨베이어/흐름 애니메이션, 디지털 트윈형 뷰 부재.**

이 갭들은 "데이터가 정확한 대시보드"에서 "현장에서 한눈에 이상을 감지하는 MES"로 넘어가는 데 필요한 요소이며, 실제 상용 MES/SCADA 제품(Ignition, AVEVA, Siemens Opcenter, MPDV HYDRA 등)의 핵심 차별화 지점이기도 하다(§3 참조).

## 2. 현재 SmartMES 시각 자산 감사

### 2.1 공통 UI 컴포넌트 (`components/ui/`)

| 컴포넌트 | 파일 | 렌더링 내용 | 애니메이션 여부 |
|---|---|---|---|
| `GaugeTile` | `components/ui/gauge-tile.tsx` | SVG 원형 게이지(`stroke-dasharray`/`stroke-dashoffset`로 0–100% 표현) | **정적.** `value`가 바뀌면 `strokeDashoffset`이 CSS transition 없이 즉시 점프. 부드러운 바늘/링 전이 없음 |
| `StatusPill` | `components/ui/status-pill.tsx` | 상태별 색상 배지(`ok/warn/crit/info/neutral`) + 작은 점(dot) | **정적.** `dot`는 고정 크기의 단색 원. `RUN/STOP/ALARM` 등 어떤 상태든 색만 다르고 움직임 없음 |
| `Sparkline` | `components/ui/sparkline.tsx` | `<polyline>` 기반 미니 추세선(52×20px) | **정적.** 값 배열을 매번 새로 그릴 뿐 애니메이션 트랜지션 없음. 실시간 갱신 시에도 즉시 리렌더 |
| `KPITile` | `components/ui/kpi-tile.tsx` | 카드형 KPI(값 + 델타 화살표 + 선택적 Sparkline) | **정적.** 값이 바뀌어도 flash/하이라이트 없음 |
| `ConnectionBadge` | `components/ui/connection-badge.tsx` | SSE 연결 상태(연결됨/재연결 중/끊김) 배지 | **유일하게 애니메이션 존재.** `reconnecting` 상태에서 Tailwind 기본 `animate-pulse` 사용 (코드베이스 전체에서 사실상 유일한 동적 시각 효과) |
| `ProgressBar` | `components/ui/progress-bar.tsx` | 진척률 막대(`width: pct%`) | **부분적.** `transition-[width]` 클래스가 있어 진행률이 바뀔 때 너비만 부드럽게 전이됨(Tailwind 기본 150ms). 색상/펄스 없음 |
| `DataTable` | `components/ui/data-table.tsx` | TanStack Table 기반 정렬/필터/페이지네이션 테이블 | **정적.** 행 추가/변경 시 하이라이트 없음 |

**결론**: 전체 UI 킷에서 실제 "움직이는" 요소는 `ConnectionBadge`의 `animate-pulse` 단 하나이며, 이마저도 SSE 연결 상태에만 쓰이고 설비/알람 상태에는 쓰이지 않는다.

공통적으로 SSE 훅(`useLiveEquipment`, `useLiveWip`, `useLiveExec`, `useLiveDashboard` — `components/app/use-live-*.ts`)은 서버가 보내는 JSON 스냅샷을 그대로 `setState`로 교체하는 구조다. 즉 **데이터 배관은 실시간이지만, 값이 바뀌었다는 사실을 사용자가 시각적으로 인지할 방법이 색상 텍스트를 다시 읽는 것뿐**이다(변경 flash, 애니메이션 없음).

### 2.3 디자인 토큰 (`lib/design-tokens.ts`, `app/globals.css`)

- 상태→색 매핑은 이미 정의되어 있다: `STATUS_COLOR.equipment = { RUN: "ok"(녹색), STOP: "neutral"(회색), REPAIR: "crit"(적색) }`. 단, 실시간 화면에서 실제 쓰이는 `RunState`는 `RUN|STOP|IDLE|ALARM` 4종(`lib/services/equipment-state-service.ts`)이며 `IDLE=warn`(노랑), `ALARM=crit`(빨강)로 매핑되어 있어 ISA-101식 상태색 관습(가동=녹/유휴=황/정지 또는 이상=적/오프라인=회)과 대체로 부합한다.
- `app/globals.css`에는 라이트/다크 테마 CSS 변수(`--bg, --surface, --ok, --warn, --crit, --info, --neutral` 등)만 정의되어 있고 **`@keyframes` 커스텀 애니메이션은 단 하나도 없다.** `tailwind.config.ts`에도 `extend.animation`/`extend.keyframes` 정의가 없어, `animate-pulse`/`animate-spin` 등 Tailwind 기본 유틸리티 외에는 애니메이션을 쓸 수 없는 상태다.
- `transitionDuration: { DEFAULT: "150ms" }`만 커스텀되어 있어 전역적으로 매우 짧은 전이만 기본 적용됨.

### 2.4 명시적으로 없는 것 (Missing)

리포지토리 전체(`app/`, `components/`)를 검색한 결과 다음 항목은 **전혀 존재하지 않는다**:
- 공장/라인 **플로어맵(평면 배치도)** — 설비를 실제 라인 위치에 맞춰 배치한 2D 도면. 현재는 모든 화면이 그리드/칸반 카드 나열뿐.
- **설비 아이콘/SVG 심볼** — CNC, 컨베이어, 로봇, 프레스 등 장비 유형별 그래픽 기호. 현재는 텍스트(`equipmentCode · equipmentName`)만 표시.
- **상태 애니메이션**: 가동 중 pulse(심장박동형 표시), 알람 blink(점멸), 값 변경 시 flash, 게이지 바늘/링의 부드러운 전이.
- **Andon 보드** — 라인 전체 상태를 대형 화면용으로 요약하는 전용 뷰(현재 화면들은 모두 데스크톱 대시보드 레이아웃이며 원거리 시인성을 고려한 대형 문자/색 블록 뷰가 없음).
- **컨베이어/흐름 애니메이션** — 자재 흐름 방향을 화살표/이동으로 표현하는 요소.
- **디지털 트윈형 뷰** — 실시간 데이터를 라인 도면 위에 오버레이하는 뷰.
- **번들 실시간 트렌드 차트 라이브러리** — `package.json`에 recharts/visx/d3/chart.js 등 차트 라이브러리가 전혀 설치되어 있지 않음. `Sparkline`은 자체 제작한 최소 SVG polyline뿐이며 히스토리 window/줌/다중 시계열 비교 기능 없음.

### 3.2 공통 패턴 요약

위 조사에서 반복적으로 확인되는 5가지 패턴:
1. **공간적 배치(플로어맵)** — 카드 나열이 아니라 실제 라인 도면 위에 설비를 배치하고 상태를 오버레이한다(Ignition, MPDV, 디지털 트윈 계열 공통).
2. **상태색 관습의 통일** — 녹=가동/황=경고·유휴/적=알람·정지/회=오프라인이 업계 공통 관습이며, SmartMES의 기존 `STATUS_COLOR` 매핑과 이미 일치한다.
3. **애니메이션은 "의미 전달"에만 사용** — ISA-101이 명시적으로 강조. 장식적 모션 금지, 이상 상태·흐름·변화만 움직인다.
4. **Andon과 대시보드는 별개의 뷰** — 원거리 시인성이 필요한 대형 화면(Andon)과 상세 KPI 대시보드는 레이아웃 요구사항이 다르다.
5. **실시간 트렌드/이력 뷰가 필수** — Sepasoft/Grafana/WinCC 모두 값의 스냅샷이 아니라 시계열 추세와 다운타임 파레토를 함께 보여준다.

### 3.3 CSP/기술스택 제약과 정합성

SmartMES의 `next.config.mjs`는 `default-src 'self'`, `connect-src 'self'`, `img-src 'self' data:`, `script-src 'self' 'unsafe-inline'`(운영 기준)의 엄격한 CSP를 이미 적용 중이다. 즉:
- 외부 CDN(Google Fonts, 외부 차트 JS CDN, 외부 SVG 스프라이트 서버 등)은 전혀 쓸 수 없다 — **모든 것은 번들(npm 의존성으로 설치 후 Next.js 빌드에 포함)되거나 인라인이어야 한다.**
- `package.json`에는 현재 차트 라이브러리가 전혀 없다(recharts/visx/d3 미설치) — 신규 트렌드 차트는 (a) 기존 `Sparkline`처럼 순수 인라인 SVG로 직접 구현하거나 (b) 가벼운 차트 라이브러리를 `npm install`로 번들링해야 한다. 둘 다 CSP상 허용되지만, 번들 크기·유지보수 관점에서 이 프로젝트 규모(사내 MES)에는 **자체 SVG 확장(sparkline → 축/툴팁 있는 미니 트렌드 차트)이 우선 권장**된다.
- 애니메이션은 외부 라이브러리(GSAP, Lottie 등, 외부 CDN 로드가 일반적) 없이 **순수 CSS(Tailwind `@keyframes` 확장) + 인라인 SVG SMIL 대신 CSS 애니메이션**으로 구현 가능하고 이것이 CSP·번들 크기 모두에 가장 안전하다.

## 5. 구현 계획

> 원칙(ISA-101 준수): 색은 이상 상태·변화 신호 전용으로 예약하고 평상시 화면은 절제한다. 애니메이션은 항상 의미를 전달할 때만 쓰고, `prefers-reduced-motion: reduce`를 존중해 모션을 비활성화하거나 대체 수단(테두리 강조 등)으로 전환한다. 모든 신규 애니메이션/색상은 기존 `lib/design-tokens.ts`의 `STATUS_COLOR`/`SEMANTIC_COLORS` 체계를 재사용하고 새 색을 추가하지 않는다.

### 5.1 상태 애니메이션 (저비용·고효과, 최우선)

- **재사용**: `STATUS_COLOR`, `Tone` 타입(`components/ui/status-pill.tsx`), 기존 `RunState`(`RUN|STOP|IDLE|ALARM`, `lib/services/equipment-state-service.ts`), `useLiveEquipment`/`useLiveWip` SSE 훅.
- **신규 필요**:
  - `app/globals.css` 또는 `tailwind.config.ts`의 `extend.keyframes`/`extend.animation`에 3종 keyframe 추가:
    - `pulse-run`(가동 중 heartbeat: 낮은 진폭의 `box-shadow`/`opacity` 맥동, 2–3초 주기, `ALARM`보다 훨씬 은은하게)
    - `blink-alarm`(알람 점멸: `opacity` 또는 `background-color` 토글, 1초 내외 — 단, ISA-101 권고에 따라 무한 반복보다 "N회 점멸 후 정지 + 지속 강조 테두리"로 완화하는 옵션 고려)
    - `flash-value`(값 변경 시 배경색이 순간 밝아졌다가 감쇠하는 단발성 트랜지션, 300–500ms)
  - 모든 keyframe에 `@media (prefers-reduced-motion: reduce) { animation: none; }` 대응 규칙 병행 작성.
  - `GaugeTile`(`components/ui/gauge-tile.tsx`)에 `transition: stroke-dashoffset 400ms ease-out` 추가(컴포넌트 내 인라인 style 또는 Tailwind `transition-all` 클래스) — 값 점프 대신 부드러운 링 이동.
  - `EquipmentCard`(`monitoring-client.tsx`)와 `WipOrderCard`(`wip-client.tsx`)에서 이전 값과 비교해 변경된 필드에 `flash-value` 클래스를 짧게 부여하는 소규모 훅(예: `useFlashOnChange(value)`) 신설 — 기존 SSE `setRows`/`setBoard` 로직은 그대로 두고 렌더 레이어에서만 diff.
  - `StatusPill`에 `tone === "crit" && animate` 같은 선택적 prop을 추가해 `ALARM`/`REPAIR` 상태일 때만 `blink-alarm` 클래스를 조건부 적용(현재 `equipmentTone`에서 `REPAIR: "crit"`로 매핑되는 지점 재사용).
- **난이도/우선순위**: 난이도 낮음(순수 CSS + 소규모 훅), **효과 대비 비용이 가장 좋은 항목** → **최우선(1순위)**.
- **ISA-101 준수 노트**: `pulse-run`은 은은하게(진폭 최소화), `blink-alarm`만 명확히 눈에 띄게 — "정상 상태는 조용히, 이상 상태만 시끄럽게"를 지켜야 한다.

### 5.2 공장/라인 플로어맵 (SVG + SSE)

- **재사용**: 기존 `EquipmentStateRow`/`RunState`/SSE 스트림(`/api/equipment/stream`), `STATUS_COLOR.equipment` 색 매핑, `useLiveEquipment` 훅(구독 로직 그대로 재사용, 렌더만 카드 그리드 대신 SVG 노드로 교체).
- **신규 필요**:
  - 작업장(`workCenter`)의 물리적 배치 좌표를 정의하는 소규모 설정 데이터(신규 테이블이 아니라 우선은 JSON/TS 상수로 좌표를 하드코딩 — 작업장 수가 적은 MVP 단계에서는 DB 마이그레이션 불필요, 필요 시 후속 R차수에서 `WorkCenter`에 `layoutX/layoutY` 컬럼 추가 검토).
  - 인라인 SVG 컴포넌트(`FloorMapView`, `components/ui` 또는 `components/app` 신설) — 라인 순서대로 배치된 사각형/아이콘 노드 + 노드 간 연결선(공정 순서), 각 노드는 `equipmentTone(row.runState)` 색으로 채우고 §5.1의 `pulse-run`/`blink-alarm` 클래스를 그대로 적용.
  - 신규 라우트 후보: `app/mockups/floor-map/` (기존 `app/mockups/monitoring`과 동일 패턴으로 `page.tsx` + `-client.tsx` 구조 재사용).
- **난이도/우선순위**: 중간(좌표 설계 + SVG 레이아웃 신규 작성, 그러나 데이터·SSE 배관은 100% 재사용) → **2순위**.
- **ISA-101 준수 노트**: 배경/배관/구조선은 회색조로 절제하고, 색은 오직 설비 상태 노드에만 적용.

### 5.3 설비 아이콘/심볼

- **재사용**: 없음(신규 시각 자산이지만 코드 로직은 없음).
- **신규 필요**: 설비 유형(가공/조립/검사/포장/컨베이어 등) 별 최소 10종 내외의 인라인 SVG 아이콘 세트 — 외부 아이콘 CDN(예: Iconify API) 대신 **직접 그리거나 `lucide-react`(이미 설치되어 있고 번들되므로 CSP에 문제 없음, `ArrowUp/ArrowDown` 등 이미 사용 중)의 산업 관련 아이콘을 우선 활용**, 부족한 것만 커스텀 SVG로 보완.
- **난이도/우선순위**: 낮음~중간(디자인 리소스 필요) → **플로어맵(§5.2)과 병행 시 가장 효율적** → **2순위와 함께 진행**.

### 5.4 Andon 보드 (대형 라인 상태 뷰)

- **재사용**: `summarizeLineTone`(`wip-client.tsx`)의 라인 대표 상태 산출 로직, `useLiveWip`/`useLiveEquipment` 데이터, `STATUS_COLOR`.
- **신규 필요**:
  - 신규 라우트(`app/mockups/andon/`) — 기존 대시보드보다 정보 밀도를 크게 낮추고 폰트/카드 크기를 대폭 확대한 전용 레이아웃(키오스크/TV 시청 거리 기준, `app/kiosk/`가 이미 존재하므로 해당 패턴·전체화면 레이아웃 관례를 참고).
  - 라인별 대형 상태 블록(전체 배경색이 상태색, §5.1의 `blink-alarm` 적용) + 최소 텍스트(라인명, 현재 상태, 정지사유).
- **난이도/우선순위**: 낮음(레이아웃 위주, 로직은 기존 것 재사용) → **3순위**.
- **ISA-101/Andon 준수 노트**: Andon 보드는 원거리 시인성이 핵심이므로 일반 대시보드의 조밀한 KPI 나열과 분리된 별도 뷰로 유지하고, 정보를 과도하게 채우지 않는다(Robato/ProjectManager 모범사례).

### 5.5 실시간 트렌드 차트

- **재사용**: `Sparkline`(`components/ui/sparkline.tsx`)의 `sparklinePoints` 좌표 변환 로직 확장, KPI/설비 이력 API(있다면) 또는 SSE로 누적되는 클라이언트 측 rolling window.
- **신규 필요**:
  - `Sparkline`을 확장한 `TrendChart` 컴포넌트 — 축 라벨, hover 툴팁(마우스 근접 지점의 값 표시), 다중 시계열(예: OEE vs 목표선) 지원. **외부 차트 CDN 없이 순수 인라인 SVG로 구현**(현재 코드베이스 스타일과 CSP 제약에 가장 부합).
  - 차트 라이브러리 번들이 필요할 정도로 요구사항이 커지면(줌/브러시 등) `recharts`를 `npm install`로 번들링 검토(런타임에 외부 스크립트를 로드하지 않으므로 CSP `script-src 'self'`와 충돌 없음).
- **난이도/우선순위**: 중간 → **4순위**(플로어맵/Andon 이후, 데이터 분석 심화 단계에서).

### 5.6 컨베이어/흐름 애니메이션

- **재사용**: §5.2 플로어맵의 연결선.
- **신규 필요**: 연결선 위에 `stroke-dasharray` + CSS `@keyframes`로 점선이 흐르는 듯한 이동 애니메이션(자재 흐름 방향 표시), WIP 수량에 따라 흐름 속도/밀도를 다르게 표현.
- **난이도/우선순위**: 낮음(순수 CSS, 플로어맵에 종속) → **플로어맵 완료 후 곧바로 추가 가능한 저비용 항목**, 단 필수도가 낮아 **5순위(선택적)**.

## 6. 권장 착수 순서

1. **① 상태 애니메이션** (§5.1) — pulse/blink/flash + 게이지 전이. 코드 변경 범위가 가장 작고(`globals.css`/`tailwind.config.ts` keyframe 추가 + 기존 컴포넌트에 클래스 조건부 적용), 사용자 체감 효과가 가장 크다. 신규 화면/라우트 불필요.
2. **② 라인 플로어맵** (§5.2, SVG + 기존 SSE 재사용) + **설비 아이콘**(§5.3, 병행). 공간적 표현이 생겨야 이후 Andon/디지털 트윈 논의가 의미를 가지므로 이 순서를 지킨다.
3. **③ Andon 보드** (§5.4) — 플로어맵/아이콘에서 만든 상태 요약 로직과 애니메이션 클래스를 그대로 재사용해 대형 화면 전용 레이아웃만 추가하면 되므로 ②의 직접적인 파생 작업.
4. **④ 실시간 트렌드 차트** (§5.5) — 데이터 심층 분석 요구가 커질 때 진행. `Sparkline` 확장으로 점진적으로 고도화.
5. **⑤ 컨베이어/흐름 애니메이션, 디지털 트윈 통합 뷰** (§4의 Low 우선순위 항목) — 필요성이 확인되면 선택적으로 추가.

각 단계는 독립적으로 배포 가능하도록 설계되어 있으며(①은 기존 컴포넌트 수정만, ②~③은 신규 라우트 추가), 기존 SSE 백엔드(`/api/equipment/stream` 등)와 `EquipmentState`/`RunState` 도메인 모델은 전 단계에서 변경 없이 재사용한다.
