# SmartMES 상용화 갭 분석 & 실운영 테스트 전략 v1.0

> 관점: MES 전문가 리뷰. 현재 코드베이스(R1~R4, 113 unit tests, Prisma+SQLite, Next14) 실측 기반.
> 결론: **기능 폐루프(수주→생산→품질→재고→구매/영업→MRP→매출/수금)와 UX/디자인은 데모/파일럿 수준으로 우수**하나, **동시성·기준정보관리·운영/보안·실시간성**에서 상용 운영 전 반드시 메워야 할 갭이 있음.

---

## A. 상용화 갭 (우선순위: P0 필수 → P1 중요 → P2 확장)

### A-1. 데이터·아키텍처 (P0)
| 갭 | 현황 | 위험 | 해결 |
|---|---|---|---|
| **SQLite 동시 쓰기 한계** | 기본 DB가 SQLite(쓰기 직렬화) | 현장 다수 작업자 동시 POP/스캔·상태전이 시 `database is locked` 경합, 처리량 상한 | 상용은 **PostgreSQL** 전환(스키마는 이미 이식 용이). SQLite는 개발/단일사용자용 |
| **낙관적 락 부재** | 상태전이에 version/락 없음 | 두 작업자가 같은 WorkOrder/Shipment/Invoice를 동시에 완료 → 이중 처리·재고 오차(race) | `@version` 필드 + `updateMany(where version)` 조건부 갱신, 실패 시 409 |
| **채번 불안정** | `Date.now()+메모리 seq`(예 PO-/INV-) | 서버 재시작 시 seq 리셋·다중 인스턴스에서 중복 코드 | DB 시퀀스/전용 counter 테이블·`@@unique`로 원자적 채번 |
| **단일 사이트/테넌트** | siteId/tenantId 없음 | 다공장·다법인 운영 불가 | 상용 요건 확정 후 scope 컬럼 + 행수준 필터/권한 |
| **하드 삭제·보존정책** | soft delete/deletedAt 없음 | 규제(이력·감사) 대응 약함 | soft delete + 보존기간 정책 |

### A-2. 기준정보 관리 MDM (P0)
- **품목·BOM·라우팅·작업장 CRUD 화면/API 부재** — 현재 seed로만 존재. `app/api`에 `items/bom/routings/work-centers` 없음.
  → 실운영은 기준정보 등록/수정/버전관리가 **1순위 필수**. (설비·거래처·모델/도면은 있음)
- BOM 유효성(순환참조·유효일자), 라우팅-공정 정합, 품목 단종/대체품 관리 필요.

### A-3. 보안·인증 (P0~P1)
| 갭 | 현황 |
|---|---|
| 입력 검증 라이브러리 없음 | 라우트마다 수동 검증 → 경계/타입/인젝션 방어 취약. **zod 스키마 검증** 도입(P0) |
| 로그인 rate-limit/계정잠금 없음 | brute-force 방어 부재(P0) |
| 비밀번호 정책·변경·초기화·만료 없음 | 최초 비번 강제변경·복잡도·이력(P1) |
| 보안 헤더(CSP·HSTS·X-Frame) / CSRF | Next 기본 외 명시적 헤더 없음(P1) |
| 권한 세분화 | 3롤(ADMIN/OPERATOR/VIEWER) 고정 — 실무는 화면·기능·필드 단위 권한 필요(P1) |
| 세션 | exp는 있음(양호). 강제 로그아웃/동시세션 제한 없음(P2) |

### A-4. 운영·관측성 (P0~P1)
- **에러 경계 부재**: `error.tsx`/`global-error.tsx`/`not-found.tsx` 없음 → 런타임 에러 시 흰 화면(P0).
- **구조적 로깅·에러추적 없음**: pino/Sentry 등 부재 → 장애 원인추적 불가(P1).
- **배포/CI 부재**: Dockerfile·docker-compose·`.github/workflows` 없음 → 재현가능 배포·회귀 게이트 없음(P0 for 운영).
- **백업/DR·운영 마이그레이션 절차** 문서/자동화 없음(P1).
- 헬스체크(`/api/health`)·메트릭 없음(P1). 감사로그는 존재(양호), 커버리지 확대 필요.

### A-5. 기능 완성도 (MES 실무) (P1)
- **실시간성 부족**: 라인 현황/대시보드/POP가 `router.refresh()`(수동, 10개 화면). 실제 현장은 **SSE/WebSocket 실시간 push**가 사실상 필수.
- **라벨/바코드 프린팅**·리포트(PDF) 출력 없음 — 현장 라벨발행은 핵심.
- **CSV/Excel**: export 2개 화면만(inventory/work-orders). 전면 확대 + **대량 가져오기(import)** 필요.
- **파일 첨부/도면 업로드**: DocumentRev는 메타만, 실제 파일 스토리지 없음.
- **알림 실제 발송**: Alarm 모델은 있으나 이메일/SMS/푸시 채널 없음.
- **분석 리포트**: 리드타임·납기준수율(OTD)·재고회전·불량 파레토 등 경영 리포트 미흡.
- **i18n·타임존**: 한국어 하드코딩. 날짜/TZ는 이미 off-by-one 발견 이력 → 전반 점검 필요.

### A-6. 성능·확장 (P1)
- **페이지네이션/서버사이드 필터·정렬 없음**: 목록 전량 로드 → 데이터 수만건 시 지연.
- **재고 파생 비용**: `deriveStock`가 전체 수불 집계 → 트랜잭션 누적 시 느려짐. **집계 스냅샷/기간마감** 테이블 필요.
- 인덱스·N+1 점검(대량 데이터 기준 EXPLAIN).

> 요약: **기능은 넓게 완성, 상용화 갭은 "많은 사용자가 동시에·오래·안전하게" 쓰는 비기능 영역에 집중.**

## B. 실운영처럼 테스트하는 방법 (이슈를 미리 잡기)

### B-0. 현 테스트 자산
- 단위 113개(도메인·서비스) 강함 ✅. 그러나 **E2E 자동화가 레포에 없음**(검증은 매번 임시 Playwright). → 가장 먼저 **E2E 스위트를 정식 커밋 + CI 게이트화**.

### B-1. 테스트 계층 (피라미드 완성)
1. **단위**(유지) → 2. **통합**(API+DB 트랜잭션·정합) → 3. **E2E**(Playwright 정식 도입, 폐루프 시나리오) → 4. **부하/동시성** → 5. **장애주입** → 6. **보안**.

### B-2. UAT 시나리오 (업무 흐름 전체, 역할별 스크립트)
실제 하루 운영을 대본화하여 순서대로 수행:
```
[생산] 기준정보 등록 → 수주 → 생산계획 → 작업지시 → 현장 POP(실적) → 재공/완제품 입고
[품질] 생체/공정/출하 검사 → 부적합/특채 → 판정
[물류] 자재 발주 → 입고(재고+) → 출하(재고−) → 반품(재고+)
[계획] MRP 순소요 확인 → 발주/생산 반영
[영업] 청구 발행 → 부분수금 → 완납 → 미수금 마감
[이상계] 취소·반품·재작업·중복스캔·권한없음·마감후정정
```
각 단계 **기대 결과 + 데이터 정합(재고=수불, 계보, 청구=수금)**을 체크리스트로.

### B-3. "실 운영 모사" — 합성 데이터 + 파일럿(가장 중요)
- **합성 데이터 생성기**: seed(수개)가 아니라 **실 규모**(품목 수천, 수불 수십만, 수주/작업지시 수천)를 생성 → 성능·UX·목록속도 체감. (스크립트 1개로 반복 가능)
- **Shadow 운영(병행)**: 파일럿 라인 1개를 기존 방식과 **동시 입력**하여 2~4주 운영, **일일 마감 수치 대조**(수량·재고·불량·출하). 불일치가 곧 이슈.
- **일일 마감 배치 감사**: 매일 정합 점검 잡 실행(아래 B-6).

### B-4. 동시성·부하 테스트 (SQLite 한계·race 노출)
- 도구: **k6**(API 부하) 또는 Playwright 병렬 워커.
- 시나리오: N명이 **동시에** ①같은 라인 POP 폭주, ②같은 오더 동시 완료, ③동시 수금/출하.
- 확인: `database is locked` 발생률, 응답 p95, **중복 채번**, **재고/상태 race(이중 차감)**. → 낙관적 락·PostgreSQL 필요성 정량 검증.

### B-5. 장애 주입(Chaos) & 멱등성
- 트랜잭션 도중 서버 재기동 → 부분 커밋/롤백 확인.
- 네트워크 단절·중복 요청(더블클릭·재시도) → **멱등키**로 이중 처리 방지 확인.
- **중복 RFID/바코드 스캔**, 동시 수금, 마감 후 소급 입력 시 방어.

### B-6. 데이터 정합 자동 감사 (불변식 검사 잡)
정기 실행해 0건이어야 하는 검사:
- 재고 = Σ수불(파생) 일치, 음수재고 없음.
- 청구액 ≥ Σ수금, 상태 = 계산상태 일치.
- 계보 무결성(고아 노드·순환 없음), 상태머신 불변식(불가능 전이 0).
- 작업지시 실적 합 ≤ 지시수량 등.

### B-7. 보안 테스트 (`/security` 스킬 활용)
- 권한 우회(수직=viewer가 변경, 수평=타 사용자 자원), 세션 위·변조, 로그인 brute-force, 입력 인젝션, 감사로그 기록 확인.

### B-8. 회귀 게이트(CI) & 수용 기준(Exit Criteria)
- PR마다 자동: **lint → unit → build → E2E 스모크**. 실패 시 머지 차단.
- 상용 전환 기준: 핵심 폐루프 E2E 100% green / P0·P1 이슈 0 / 성능 SLA(p95 목표) 충족 / 정합 감사 0 불일치 / 보안 항목 통과.

## C. 권장 착수 순서

**P0 (상용화 전 필수)**
1. 정식 **E2E 스위트 + CI**(회귀 안전망 먼저) + **합성 데이터 생성기**
2. **기준정보 CRUD**(품목·BOM·라우팅·작업장)
3. **입력검증(zod)** + **error 경계** + **로그인 rate-limit**
4. **낙관적 락 + 채번 안정화**, 그리고 **PostgreSQL 전환 옵션**(부하테스트 결과에 따라)

**P1**: 실시간(SSE)·로깅/Sentry·페이지네이션·보안헤더·백업/Docker·리포트/CSV 확대·라벨 프린팅
**P2**: 다중 사이트/테넌트·i18n·파일 업로드·권한 세분화

> 제안: 먼저 **B-0/B-1의 E2E 스위트+CI와 B-3 합성 데이터 생성기**를 세워 "실 운영하듯" 반복 테스트 기반을 만든 뒤, 거기서 나오는 이슈로 P0를 우선순위대로 처리하는 것이 가장 효율적.
